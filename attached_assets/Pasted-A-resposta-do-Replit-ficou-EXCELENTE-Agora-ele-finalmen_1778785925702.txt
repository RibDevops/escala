# AJUSTES IMPORTANTES

---

# 1. O TERMO "COLUNA MAIS VAZIA"

O texto ainda mistura:

```text id="n9v2q0"
coluna mais vazia
```

com:

```text id="jlwm84"
menor quantidade de serviços
```

Agora sabemos que a regra REAL é:

```text id="jlwm3o"
menor quantidade total de serviços reais
```

A coluna NÃO é o critério principal.

Ela apenas representa:

* progressão histórica visual

Sugestão:
trocar toda referência de:

* “coluna mais vazia”

por:

* “militar com menor quantidade de serviços”

---

# 2. FOLGA NÃO OCUPA COLUNA REAL

Esse é MUITO importante.

Hoje o exemplo visual dá impressão que:

```text id="jlwm0w"
FOLGA ocupa coluna histórica
```

Mas tecnicamente:

* NÃO ocupa coluna persistida
* NÃO aumenta count
* NÃO cria serviço

Ela existe SOMENTE:

```text id="jlwm0q"
na matriz operacional temporária
```

Sugestão:

Separar claramente:

## MATRIZ HISTÓRICA

Somente serviços reais.

## MATRIZ OPERACIONAL

Serviços + bloqueios temporários.

---

# 3. O FALLBACK PRECISA SER MAIS CONTROLADO

Esse ponto:

```text id="jlwm2s"
“militar com folga mais curta”
```

pode quebrar regras operacionais críticas.

Sugestão mais segura:

## Ordem:

1. tentar todos válidos
2. gerar alerta
3. tentar relaxamento configurável
4. somente então fallback

Porque:

* férias nunca devem ser quebradas
* indisponibilidade médica nunca deve ser quebrada

---

# 4. O ALGORITMO NÃO PROCURA "PRÓXIMO SERVIÇO"

Ele procura:

```text id="jlwm1r"
próximo militar válido
```

Isso parece pequeno mas muda:

* nomenclatura
* clareza
* manutenção

---

# 5. A MATRIZ OPERACIONAL É A PEÇA MAIS IMPORTANTE

Esse ponto precisa ficar mais forte na arquitetura.

Porque:

* ela é o cérebro real do sistema
* o banco é apenas persistência histórica

A geração inteira acontece:

```text id="jlwm5l"
em memória
```

igual o escalante faria visualmente no Excel.
